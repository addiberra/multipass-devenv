== Technical Context

This feature creates an isolated development environment for agentic coding tools by launching a fresh Multipass VM and provisioning it with cloud-init. The simplified design now relies on VM isolation as the primary boundary rather than in-guest filesystem MAC controls or a project-managed teardown workflow.

Key security principles:

* **Kernel isolation**: Full VM virtualization separates agent execution from the host kernel.
* **No host mounts**: Repositories are cloned inside the VM instead of mounted from the host.
* **Network egress controls**: iptables applies a default-deny outbound policy with a small allowlist.
* **Secret injection**: Credentials are written into the VM by cloud-init instead of inherited from the host environment.
* **Non-root execution**: OpenCode runs as the `opencode` user with no administrative privileges.
* **Domain-level filtering**: Whitelist domains are resolved to IP addresses at launch time for iptables enforcement.

== Decisions and Constraints

* ADR-0001: Use Multipass VM for Secure Agentic AI Development Environment (see `docs/adrs/0001-multipass-sandbox.yaml`)

Relevant constraints from `docs/constraints.yaml`:

* C-001: No host directory mounts
* C-002: Network egress blocked except whitelisted domains
* C-003: Secret injection via cloud-init write_files
* C-004: SSH disabled
* C-005: Non-root user execution
* C-008: No host credential exposure
* C-010-C-016: Supply chain integrity controls
* C-017-C-019: DNS and network restrictions

=== Host Components

==== launch.sh

`scripts/launch.sh` is the only project-managed entrypoint. It:

* parses `--repo`, `--secrets`, `--network`, `--name`, and `--dry-run`
* validates the secrets file and optional network whitelist
* reads `cloud-init/base.yaml` and `cloud-init/security.yaml` as source YAML rather than duplicating their content inline
* parses the network whitelist YAML (default or custom) to extract allowed domains, ports, and DNS servers
* resolves each whitelisted domain to IP addresses using `dig +short` on the host at launch time
* generates per-IP iptables rules for each resolved address and port combination
* generates DNS iptables rules from the whitelist's `dns_servers` field
* appends a runcmd step that removes the `opencode` user from the `sudo` group after all setup commands complete
* merges the above into a single cloud-init document
* launches the VM through Multipass
* reports the VM name and workspace path

==== config/network-whitelist.default.yaml

Defines the default outbound allowlist for git providers, LLM APIs, package registries, and trusted DNS resolvers. The file contains three sections consumed by `launch.sh`:

* `egress_allowed` -- list of domain/port pairs that `launch.sh` resolves to IPs for iptables rules
* `dns_servers` -- list of resolver IPs used for iptables DNS allow rules and domain resolution at launch time

=== VM Components

==== Base Provisioning (`cloud-init/base.yaml`)

The base layer installs packages from official Ubuntu repositories, creates the `opencode` user, and prepares the workspace, config, and `.ssh` directories.

==== Security Provisioning (`cloud-init/security.yaml`)

The security layer provides the static runcmd skeleton:

* applies a default-deny outbound iptables policy (OUTPUT, INPUT, FORWARD all DROP)
* allows loopback traffic and established/related connections
* disables SSH daemon access so the VM is entered through `multipass shell`
* locks the OpenCode config directory after secret injection

`launch.sh` injects additional runcmd entries generated at launch time:

* per-IP iptables rules for each resolved whitelist domain and port (e.g., `iptables -A OUTPUT -p tcp -d 140.82.121.4 --dport 443 -j ACCEPT`)
* DNS resolver rules derived from the whitelist's `dns_servers` field
* a final step that revokes sudo privileges from the `opencode` user (`deluser opencode sudo`)

This separation keeps `security.yaml` as a stable base while `launch.sh` generates the dynamic, environment-specific rules.

==== Domain Resolution

At launch time on the host, `launch.sh` resolves each domain in the network whitelist to one or more IP addresses using `dig +short` (falling back to `getent ahosts` when `dig` is unavailable). For each resolved IP and each allowed port, an iptables OUTPUT ACCEPT rule is generated and injected into the cloud-init runcmd section. If a domain cannot be resolved, `launch.sh` logs a warning and skips that domain.

This approach has a known limitation: IP addresses may change after resolution. For long-lived VMs, the resolved IPs may become stale. This is an acceptable trade-off because the VMs are intended to be ephemeral.

==== OpenCode Installation

OpenCode is downloaded from the official release source, verified against a pinned SHA256 checksum, installed into `/usr/local/bin`, and validated with `opencode --version`.

==== Secret Injection

`launch.sh` reads the developer-provided secrets file and renders cloud-init `write_files` entries for:

* `/home/opencode/.config/opencode/config.yaml`
* `/home/opencode/.gitconfig`
* `/home/opencode/.ssh/known_hosts`

Secrets exist only inside the VM after provisioning.

== Requirement Resolution

=== US-001: Launch Secure Dev Environment

Components:: `scripts/launch.sh`, Multipass, `cloud-init/base.yaml`
Requirements::
* DEV-001: `launch.sh` invokes `multipass launch` with Ubuntu 24.04.
* DEV-002: cloud-init creates the non-root `opencode` user.
* DEV-003: `--dry-run` prints generated configuration without launching a VM.
* DEV-004: launch aborts if the secrets file is missing or unreadable.

=== US-002: Configure OpenCode with Secrets

Components:: `scripts/launch.sh`, `cloud-init/security.yaml`, secrets file
Requirements::
* DEV-005: cloud-init installs OpenCode from the official source.
* DEV-006: installation includes SHA256 verification before placement in `/usr/local/bin`.
* DEV-007: secrets are written into the VM through cloud-init `write_files`.
* DEV-008: the VM starts from a clean environment instead of inheriting host variables.

=== US-003: Work on Git Repositories

Components:: `scripts/launch.sh`, `cloud-init/base.yaml`, `cloud-init/security.yaml`
Requirements::
* DEV-009: `launch.sh` clones the requested repository into `/home/opencode/workspace`.
* DEV-010: GitHub and GitLab host keys are pre-seeded in `known_hosts`.
* DEV-011: git user name and email come from the secrets file.

=== US-004: Restrict Network Access

Components:: `scripts/launch.sh`, `config/network-whitelist.default.yaml`, `cloud-init/security.yaml`
Requirements::
* DEV-012: `launch.sh` resolves whitelist domains to IPs and generates per-IP iptables ACCEPT rules. All other outbound traffic is dropped by default.
* DEV-013: GitHub and GitLab domains are resolved and allowed on ports 22 and 443.
* DEV-014: OpenAI and OpenCode API domains are resolved and allowed on port 443.
* DEV-015: package registry domains are resolved and allowed on port 443.
* DEV-016: `--network` replaces the default allowlist including its domains, ports, and DNS servers. `launch.sh` reads `dns_servers` from the custom file and generates DNS iptables rules accordingly.
* DEV-017: malformed network config aborts launch before VM creation.

=== US-005: Verify Supply Chain Integrity

Components:: `cloud-init/base.yaml`, `cloud-init/security.yaml`
Requirements::
* DEV-018: apt packages come only from official Ubuntu repositories.
* DEV-019: Python package installation supports hash-verified requirements.
* DEV-020: OpenCode installation validates the pinned checksum before install.
* DEV-021: checksum mismatch aborts initialization.

=== US-006: Harden Runtime Privileges

Components:: `scripts/launch.sh`, `cloud-init/base.yaml`
Requirements::
* DEV-022: The final runcmd step generated by `launch.sh` runs `deluser opencode sudo` to remove the `opencode` user from the sudo group. The user retains sudo during cloud-init execution (needed for package installation, iptables, service management) but loses it before the VM is handed to the developer.
