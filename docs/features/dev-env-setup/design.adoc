== Technical Context

This feature provisions an isolated development environment for agentic coding
work by creating a Multipass VM from a project-local `devvm.yaml` file. The
prior design centered on a flag-driven Bash script that generated cloud-init
content dynamically. The new design keeps Multipass and static cloud-init as
the base hardening layer, but moves project-specific setup into a thin Go CLI
that performs post-boot provisioning with `multipass transfer` and
`multipass exec`.

The intent is to keep startup simple without weakening the existing security
model:

* Multipass remains the primary isolation boundary.
* Host directories are never mounted into the guest.
* SSH remains disabled.
* The guest runtime remains non-root.
* UFW still enforces a default-deny network posture with explicit allow rules.
* Only scoped provisioning files are delivered into the guest during setup.

== Decisions and Constraints

* ADR-0002: Use a Go Create CLI for Config-Driven Multipass Startup (see `docs/adrs/0002-go-create-cli-for-config-driven-multipass-startup.yaml`)
* ADR-0001: Use Multipass VM for Secure Agentic AI Development Environment (superseded, historical context only)

Relevant constraints from `docs/constraints.yaml`:

* C-001: No host directory mounts
* C-002: Network egress restricted to essential ports via UFW
* C-003: Secrets delivered through scoped provisioning files, not host env vars or mounts
* C-004: SSH disabled
* C-005: Non-root user execution
* C-008: No host credential exposure
* C-010 through C-016: Supply chain integrity controls

== Research and Unknowns

Research during this transition found that the remaining complexity was mostly
in host-side orchestration, not in Multipass itself. The design therefore uses
Multipass-native primitives directly:

* `multipass launch` for fresh instances
* `multipass clone` for optional base-instance reuse
* `multipass transfer` for scoped file delivery
* `multipass exec` for post-boot provisioning

The CLI is intentionally narrow:

* one user-facing command: `create`
* one project config file: `devvm.yaml`
* no general configuration-management behavior

No unresolved research unknowns remain for implementation. The main open point
for future evolution is whether a reusable base instance should be built and
managed by the same tool or kept as a manual operational step.

=== Host Components

==== `devvm.yaml`

`devvm.yaml` is the only required project input. It defines:

* instance sizing and optional naming
* repository source and optional branch
* optional base-instance preference
* guest paths and user
* provisioning file locations
* security settings such as SSH disablement and allowed outbound ports
* optional OpenCode config template location

The CLI resolves relative paths from the config file directory so a project can
remain self-contained.

==== Create CLI

The Go CLI is a thin orchestrator, not a provisioning framework. Internally it
is split into these responsibilities:

* `config`: load and validate `devvm.yaml`
* `source resolution`: decide clone vs fresh launch
* `multipass`: run `launch`, `clone`, `transfer`, and `exec`
* `provisioning`: stage the exact files required for guest setup
* `verification`: confirm the environment is ready before success is reported

The command surface stays intentionally small. The only user-facing operation is
`create`.

==== Provisioning Inputs

The CLI reads three classes of host-side inputs:

* `secrets.env` for runtime secrets and git identity
* static guest bootstrap script for per-project setup
* optional OpenCode config template for rendered guest config

These files are validated before any VM creation begins when possible.

=== Guest Components

==== Static cloud-init

`cloud-init/base.yaml` remains static and is responsible only for base guest
hardening and readiness:

* install packages from official Ubuntu repositories
* create the non-root guest user
* prepare guest directories
* configure and enable UFW with default-deny rules and allowed outbound ports
* disable SSH access

Project-specific repository and secret material is no longer embedded into
generated cloud-init content.

==== Guest provisioning flow

After the guest is ready, the CLI uses `multipass transfer` to copy only the
files required for the session into the guest. A subsequent `multipass exec`
invocation performs setup tasks such as:

* render and place OpenCode config
* write git config
* seed known hosts if needed
* clone the configured repository into `guest.repo_dir`
* check out the configured branch if present
* install or verify OpenCode in the guest
* revoke any temporary administrative privileges from the runtime user

This two-phase approach keeps base VM creation stable while allowing project
setup to vary by config.

=== Config Schema

The config contract is represented by `devvm.yaml` and validated against a
schema. The top-level sections are:

* `instance`
* `source`
* `base`
* `guest`
* `provisioning`
* `secrets`
* `security`
* `opencode`

Validation handled by the CLI includes:

* schema conformance
* required file existence for referenced host paths
* valid Multipass-compatible instance names
* positive resource sizing values
* `guest.repo_dir` contained within `guest.workspace_dir`
* `base.instance_name` present when `base.enabled` is true

=== Failure Handling

The CLI stops on the first failed precondition that would otherwise create an
invalid or half-configured environment.

Failure classes:

* config missing or invalid before VM creation
* required host file missing before VM creation
* base instance unavailable, causing logged fallback to fresh launch
* file transfer failure after VM creation
* bootstrap or repository clone failure after VM creation
* checksum validation failure for OpenCode installation

Post-creation failures are surfaced directly with the VM name so the developer
can inspect or remove the instance manually.

== Requirement Resolution

=== US-001: Create Environment from Project Config

Components:: `devvm.yaml`, create CLI config loader, Multipass adapter
Requirements::
* DEV-001: The create CLI reads `devvm.yaml` and uses its values to drive VM creation.
* DEV-002: Preflight validation rejects missing or unreadable config before invoking Multipass.
* DEV-003: Schema validation reports the invalid field and aborts before VM creation.
* DEV-004: The source-resolution step derives a valid instance name from the repository when `instance.name` is absent.

=== US-002: Choose the Startup Source Automatically

Components:: source resolution logic, Multipass adapter
Requirements::
* DEV-005: The source resolver selects `multipass clone` when a configured base instance is available.
* DEV-006: The source resolver falls back to `multipass launch` with the configured Ubuntu release and reports that fallback.

=== US-003: Provision the Guest with Scoped Inputs

Components:: provisioning input loader, transfer stage, bootstrap execution, repository setup
Requirements::
* DEV-007: Provisioning delivers only configured files into the guest and does not inherit host environment variables.
* DEV-008: Preflight validation rejects missing or unreadable secrets input before VM creation.
* DEV-009: The template-rendering and transfer path creates the OpenCode config file at the configured guest destination.
* DEV-010: The bootstrap execution stage clones the configured repository into `guest.repo_dir`.
* DEV-011: The bootstrap execution stage checks out the configured branch when one is provided.

=== US-004: Preserve Secure Runtime Defaults

Components:: static cloud-init, bootstrap execution, guest verification
Requirements::
* DEV-012: Static cloud-init and guest verification ensure the runtime user matches `guest.user` and is non-root.
* DEV-013: The final bootstrap hardening step leaves the runtime user without administrative privileges.
* DEV-014: Static cloud-init configures firewall defaults to deny inbound and outbound traffic.
* DEV-015: Firewall provisioning applies allow rules only for ports listed in `security.allowed_outbound_ports`.
* DEV-016: Static cloud-init disables SSH while preserving `multipass shell` access.

=== US-005: Verify OpenCode Supply Chain Integrity

Components:: bootstrap execution, OpenCode installer, guest verification
Requirements::
* DEV-017: OpenCode installation completes before the CLI reports success.
* DEV-018: The installer verifies the expected checksum before install.
* DEV-019: A checksum mismatch aborts setup and reports a failure.
