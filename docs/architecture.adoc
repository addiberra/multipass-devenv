== Overview

Secure Multipass development environment for agentic AI coding tools. Provides kernel-level isolation between the host and agent execution environment through full virtualization. The system creates ephemeral Ubuntu VMs with strict network and filesystem controls, injects only necessary secrets, and destroys all artifacts after session completion.

Key design principles from NVIDIA AI Red Team guidance:

* **Kernel isolation**: Full VM virtualization (not containers) separates agent execution from host kernel
* **Network egress controls**: iptables whitelist restricts outbound traffic to essential endpoints
* **Filesystem isolation**: AppArmor confines writes to designated workspace
* **Secret injection**: Credentials injected via cloud-init, not inherited from host environment
* **Ephemeral lifecycle**: VM destroyed after session to prevent accumulation of secrets/IP

== Security Architecture

=== Network Egress Controls

iptables configured with default-deny policy. Only explicitly whitelisted domains/ports permitted:

[cols="1,1,1", options="header"]
|===
| Domain | Ports | Purpose

| github.com
| 22, 443
| Git SSH/HTTPS operations

| gitlab.com
| 22, 443
| Git SSH/HTTPS operations

| api.openai.com
| 443
| OpenAI API

| api.opencode.ai
| 443
| OpenCode API

| pypi.org
| 443
| Python packages

| registry.npmjs.org
| 443
| npm packages

| crates.io
| 443
| Rust crates
|===

DNS resolution restricted to trusted resolvers (8.8.8.8, 1.1.1.1).

=== Filesystem Isolation

AppArmor profile restricts write operations:

[cols="2,1", options="header"]
|===
| Path | Permission

| /home/opencode/workspace/** 
| rw (read/write)

| /home/opencode/.config/opencode/
| r (read only, secrets injected)

| /home/opencode/
| r (read only)

| /
| deny
|===

=== Secret Injection Flow

1. Developer provides secrets file (not env vars, not mounted host files)
2. launch.sh reads secrets and passes to cloud-init
3. cloud-init write_files places secrets in VM during boot
4. Secrets live only in VM memory/filesystem
5. teardown.sh destroys VM, secrets destroyed with it

No host filesystem mounts, no environment variable inheritance.

=== Lifecycle Management

Ephemeral by design:

1. Launch: Fresh Ubuntu image each session
2. Provision: cloud-init applies security controls, installs tools
3. Work: Developer uses `multipass shell` to interact
4. Teardown: VM destroyed, all artifacts deleted

No persistent state accumulation across sessions.

== Cross-Cutting Concerns

=== Error Handling

* Network connection failures: Clear error messages when whitelisted endpoints unreachable
* Invalid secrets file: Validation before VM launch
*VM launch failures: Multipass error passthrough to developer
* Checksum verification failures: Abort OpenCode installation

=== Supply Chain Security

* All apt packages from official Ubuntu repositories only
* OpenCode downloaded from official source with SHA256 verification
* Python packages with hash verification
* npm packages with integrity verification
* All dependencies pinned to specific versions

=== Observability

* Multipass logs available via `multipass logs <instance>`
* VM console output during provisioning
* No persistent logging by design (ephemeral lifecycle)
