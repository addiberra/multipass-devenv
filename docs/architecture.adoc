== Overview

This project provisions isolated Ubuntu VMs with Multipass for agentic development work. The architecture now treats the VM boundary itself as the primary security control. Host files are never mounted into the guest, secrets are injected during provisioning, SSH is disabled, and outbound network traffic is restricted to a small allowlist.

== Security Architecture

=== Isolation Boundary

The security model depends on the Multipass VM boundary, not on in-guest filesystem MAC controls. The host and guest remain separated by virtualization, with repositories cloned directly into the guest workspace and no host credential directories mounted into the VM.

=== Network Egress Controls

iptables applies a default-deny outbound policy and permits only:

* GitHub and GitLab on ports 22 and 443
* OpenAI and OpenCode APIs on port 443
* Package registries such as PyPI, npm, and crates.io on port 443
* Trusted DNS resolvers on port 53

=== Secret Injection Flow

1. The developer provides a `secrets.env` file on the host.
2. `scripts/launch.sh` converts the values into cloud-init `write_files` entries.
3. cloud-init writes the config into `/home/opencode/.config/opencode/` and related files.
4. The config directory is locked after provisioning.

=== Access Model

The VM is entered through `multipass shell`. SSH is disabled during provisioning. The `opencode` user is the default execution identity for agent tooling.

== Cross-Cutting Concerns

=== Error Handling

* Invalid secrets or network config abort launch before VM creation.
* Multipass launch failures are surfaced directly to the developer.
* Checksum verification failures abort OpenCode installation.

=== Supply Chain Security

* apt packages come from official Ubuntu repositories only
* Python requirements are expected to use hash verification where applicable
* npm integrity checks rely on lockfile metadata
* OpenCode is downloaded from the official source with a pinned checksum

=== Observability

* `multipass info <instance>` shows VM state
* `/var/log/cloud-init-output.log` captures provisioning output inside the guest
* dropped packets are logged through iptables log rules
